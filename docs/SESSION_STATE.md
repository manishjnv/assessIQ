**Why this design (the "considered and rejected" list per DoD detail rule):**

- *Single firehose `app.log` instead of 6 streams* — rejected: `jq` on a 30 MB/day mixed file is slow and noisy; the `error.log` mirror is the killer feature, but only valuable when there are streams to mirror from. Going firehose → splits later requires touching every module's logger call; going splits → firehose later is a `cat *.log > app.log` shell one-liner. Asymmetric refactor cost favors splits now.
- *Loki / Promtail / Vector aggregator now* — rejected: 200 MB resident on the shared VPS, 2 extra containers, single point of failure. File-based + `jq` is faster to triage today on a single VPS with one ops human. Phase 3 can scrape the same JSONL files via Promtail without app-code change. Documented in `docs/11-observability.md § 11`.
- *OpenTelemetry tracing now* — rejected: listed as 00-core open question; deferred to Phase 3. `requestId` correlation handles 90% of triage today.
- *Pino multi-transport workers (one worker per file target) instead of `pino.multistream`* — rejected: at 50 req/min peak, synchronous writes via `pino.multistream` are perfectly adequate and avoid worker-thread complexity. Pino docs explicitly support this pattern.
- *Inline value-pattern sanitizer in the `_log` ingest endpoint* — rejected for Phase 0; documented as known gap. The frontend's substring sanitizer + the server's `LOG_REDACT_PATHS` cover most cases. Phase 3 adds JWT/sk-*-shaped value detection.
